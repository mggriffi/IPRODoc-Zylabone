Synchronization with Exchange

Prerequisites

  • We collect items only from the Journal mailbox, which means that:
    • All private user items will not be collected (e.g. private appointments, private tasks, drafts)
    • We don't track user changes on existing items, so we treat them equal to the original item (including all existing copies)
    • The Journal account is used to connect to a user mailbox (via impersonation), so the Journal account should have the right to impersonate every trackable mailbox
  • We track the status of user items by checking only the Recoverable Items folder, so it is a must that synchronization is run more often than the retention policy is applied (or we will not be up to date).
  • Deletion flow:
    • Litigation Hold and Single Item Recovery features by MS Exchange prevent items from being deleted from Exchange Service. They should be disabled when emails collected from Exchange are deleted from Legal Review. 
    • When we delete a meeting item, meeting cancellations are sent to the appropriate users.
    • When collecting from the Journal mailbox, we (by default) delete items. This can be set up to leave them in place.
    • When deleting items from Legal Review, we don't delete them from the Journal mailbox.

      Deleting meetings
      Exchange has a special type of calendar item called "Appointment".

      - Appointment items reside only in the user mailbox and are never sent.

      When a Meeting Request/Response (accept/decline/tentative/new time proposal) is sent, a special Email item is created, which contains information about the Appointment.

      - When the appointment is updated (time change, text, attendees, etc.), a new version of the appointment is created, while the old one is moved to the Recoverable folders (RecoverableRoot in Exchange2010, RecoverableRoot\Calendar Logging in Exchange2013).

      What this means for Legal Review and Exchange synchronization:

      - As we collect from the Journal mailbox only, we have no idea about the Appointment item. So we treat the MeetingRequest and CancellationRequest items as "Request Email" + Appointment itself. This means that when a user requests deletion of MeetingRequest/CancellationRequest email in Legal Review (you can't see it as MeetingRequest, but as a simple email), we'll also delete the appropriate version of the appointment. This is the same for the reporting Deleting/DeletedByUser status, we'll report them only when all related items are deleted from the mailbox.

      When a meeting is updated:
      - The old version of the appointment is deleted
      - The new version of the appointment is created 
      - The new email is sent

      This means that in Legal Review several versions of the same Meeting Request (but with different data inside) will be present. When in Legal Review a Meeting Request is requested to be deleted, only the appropriate version of the appointment will be deleted.

      Tasks are not deleted

      Exchange has a special item type called "Task", which can be delegated to another user. Appropriate emails (delegation request/response/update) will be captured in the Journal mailbox and collected in Legal Review.

      However, as the task itself is never sent, it won't be deleted when these emails are deleted from Legal Review.

  • Working with matter in Legal Processing/Legal Review:
    • Only one Exchange matter is allowed
    • When a matter is deleted in Legal Review 
      • Items that were previously collected in Legal Review, and that are still present at the ZyLAB Exchange Service will be reset to the state from which they can be collected again
      • Items that were deleted and reported to Legal Review are no longer present at the ZyLAB Exchange Service and won't be available for recollection

        Synchronization schedule

        When determining a schedule for the synchronization, consider other processes that compete for Mailbox server resources, such as offline defragmentation of the mailbox database and antivirus scans.

Warnings

  • A user may use Local Archive* (LA) as well as Personal Archive Mailbox** (PAM). LA is not available via Exchange, so we never know which data is stored on the user machine. Deletion from Legal Review allows to delete appropriate data on the Exchange Service, while some copies still might be stored in LA. Later, a user may re-import data from LA to PAM, which would result in a mismatch between the state in Legal Review and the data on the Exchange Server.
  • A Personal Archive Mailbox might be disabled for some period. In this case items stored in the archive are no longer available via Exchange, while they are still present until Retention Policy deletes them. A disabled archive might be re-enabled, which would make items available via the Exchange Server again. This issue might lead to a mismatch between the results in Legal Review and in the Exchange Server.
  • If a user leaves an organization and his mailbox is preserved for eDiscovery, we might not be able to access his mailbox, which might lead to an inconsistent state of ExchangeExistenceStatus and corresponding deletions from Legal Review.
  • There are some cases, when MS Exchange Server doesn't return items when a search is performed****. These items can't be accessed by the ZyLAB Exchange Service, which might lead to an inconsistent state of the ExchangeExistenceStatus and corresponding deletions from Legal Review.
  • When an email is sent "on behalf of", in the Journal mailbox this information might be missing, which would lead to missing track of the item in the original sender's mailbox.
  • In case an action failed during processing in Legal Processing, after which the item was removed from Legal Review***, then if the action is restarted in Legal Processing and succeeds - it will be added to Legal Review again with the original ExchangeExistenceStatus. 

* Local Archive - archive stored on the local machine of the user. Data is stored as .pst file. Data is not accessible via Exchange Server.

** Personal Archive Mailbox - archive stored on Exchange Server/in the cloud. Data is accessible via Exchange Server.

*** This can happen when an items family was requested to be deleted in Legal Review, and then the original item in Exchange was deleted and synchronized to Legal Review.

**** Four cases where an item cannot be found:

  1. Item is filtered by mailbox search filter
  2. Item is in a safe list
  3. Item is encrypted
  4. IRM-protected item

For more information about the four cases listed above, please see http://technet.microsoft.com/en-us/library/bb232132(v=exchg.141).aspx#ESA

Configuration

ZyLAB Exchange Service requires at least one Journal mailbox configuration to be set up in ZyLAB.EDiscovery.Exchange.Standalone.WinService.exe.config file.
<ExchangeCollectionConfiguration><Mailboxes><JournalMailbox /></ Mailboxes></ ExchangeCollectionConfiguration>

Available settings:

Exchange connection setting

  • ExchangeWebServerUri - URI to exchange service EWS
  • ExchangeVersion - version of Exchange server installed (Exchange2010, Exchange2010_SP1, Exchange2010_SP2, Exchange2013)

 

Journal mailbox connection settings

  • EmailAccount - Journal mailbox account
  • Login - login used for the Journal mailbox account; specify if it is different to the EmailAccount
  • Domain - domain; specify if accessing is different to the machines defaultPassword - Journal mailbox password

 

Journal mailbox collecting behavior

  • DeleteItemsInJournalMailbox - set to false if you do not want ZyLAB ExchangeService to remove collected items from journal mailbox; true by default

Resolution of trackable mailboxes (only one of next three values is allowed, case insensitive)

  • TrackableDomains - user mailboxes tracking will be done for domains specified here; you can specify several values separated with ";"
  • TrackableMailboxes - only user mailboxes specified in this list will be tracked
  • TrackableRegexp - trackable mailboxes will be determined based on regexp result
